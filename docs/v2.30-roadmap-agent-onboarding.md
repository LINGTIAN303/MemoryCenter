# v2.30+ 路线图：Agent 客户端无感接入

> **目标**：让 Hippocampus 在 Agent 客户端（Trae / Cursor / Claude Code / Codex 等）中
> 被注册后，LLM 无需用户提示即可主动调用记忆工具，并按客户端特性自动适配工作流。
>
> **起点**：Trae（已接入，当前 description=null，LLM 无感）
> **终点**：覆盖 4 主流 Agent 客户端 + 7 待补，每端都有 onboarding 模板与自动识别

---

## 背景与问题

### 当前痛点（v2.29 及之前）

1. **MCP server 元数据缺失**：`SERVER_METADATA.json` 的 `description` 字段为 `null`，
   LLM 看到的只是 11 个分散工具，不知道 hippocampus 是什么、何时该用
2. **工具描述是静态操作说明**：每个 tool 描述了「做什么」，但没说明「何时做」
   （如 `prompt` 工具应「会话开始时必调」，但 LLM 不会主动执行）
3. **preset 仅影响归档数值**：v2.29 的 CombinedProfile 只携带 archive_threshold +
   summary_template 等数值字段，未输出 LLM 可读的行为契约
4. **Agent 客户端无自动识别**：MCP server 启动时不知道自己被哪个客户端拉起，
   无法按客户端特性（窗口阈值/压缩方式/scenario）自动适配
5. **接入门槛高**：每个客户端的 MCP 配置格式不同，用户需手动配置环境变量与
   session_id 约定，缺少 onboarding 模板

### v2.29 的偏离与 v2.30 的修正

| 维度 | v2.29 实际产出 | v2.30 修正方向 |
|------|---------------|---------------|
| Preset 是什么 | 数值参数组合 | **行为契约模板**（含使用协议/触发规则） |
| Preset 怎么用 | archive 时传 preset 字段 | Agent 客户端接入时声明，启动时自动应用 |
| Preset 输出 | CombinedProfile（Rust struct） | LLM 可读的 instructions 文本 |
| LLM 感知 | 完全无感 | 启动即看到使用协议，按规则主动调用 |

## 任务路线图

### 阶段 1：Trae 接入闭环（v2.30，起点）

**目标**：让 Trae 中的 LLM 启动即知道 hippocampus 存在 + 主动调用 + Trae 客户端自动识别

#### 1.1 AgentFamily 加识别指纹（hippocampus-agents crate）

- [ ] 新增 `AgentFingerprint` 结构体
  ```rust
  pub struct AgentFingerprint {
      /// MCP client_info.name 匹配关键词（小写）
      pub client_info_keywords: &'static [&'static str],
      /// 父进程名匹配关键词
      pub parent_process_keywords: &'static [&'static str],
      /// 环境变量指纹（变量名前缀）
      pub env_var_prefixes: &'static [&'static str],
  }
  ```
- [ ] `AgentFamily::fingerprint()` 方法，4 主流 family 填充指纹数据
  - ClaudeCode: client_info=`["claude-code"]`, env=`["CLAUDE_CODE_"]`
  - Cursor: client_info=`["cursor"]`, env=`["CURSOR_"]`
  - Trae: client_info=`["trae"]`, env=`["TRAE_"]`
  - Codex: client_info=`["codex"]`, env=`["CODEX_"]`
- [ ] 单元测试：4 主流 family 指纹非空 + Custom 返回 generic

#### 1.2 自动识别函数（hippocampus-presets crate 新增 `detect.rs`）

- [ ] 新增 `detect_agent_client()` 函数，3 层信号融合：
  - Layer 1: 显式声明（`HIPPOCAMPUS_PRESET_AGENT` 环境变量，最高优先级）
  - Layer 2: MCP 协议信号（rmcp ClientInfo，需 rmcp 升级支持）
  - Layer 3: 进程环境指纹（父进程名 + 环境变量前缀，纯 Rust 可实现）
  - Layer 4: 未识别 → 降级为 Generic（不报错，仅缺少 preset 优化）
- [ ] 单元测试：模拟环境变量 + 进程名，验证识别结果

#### 1.3 CombinedProfile 加 `usage_protocol`（hippocampus-presets crate）

- [ ] 新增 `UsageProtocol` 结构体
  ```rust
  pub struct UsageProtocol {
      /// 注入 MCP instructions 的文本（给 LLM 看的使用须知）
      pub instructions: String,
      /// session_id 派生规则
      pub session_id_pattern: String,
      /// 工具触发规则（LLM 可读的"何时调什么工具"）
      pub trigger_rules: Vec<TriggerRule>,
  }

  pub struct TriggerRule {
      pub condition: String,
      pub tool: String,
      pub example: Option<String>,
  }
  ```
- [ ] `PresetBuilder::build()` 时根据 agent + scenario 自动生成 `usage_protocol`
- [ ] 示例：Claude Code + Coding 生成「180K 阈值 + 代码改动摘要 + session_id=claude-code-{date}」

#### 1.4 MCP server 启动时识别 + 注入 preset（hippocampus-mcp crate）

- [ ] `main.rs` 启动流程改造：
  1. 调用 `detect_agent_client()` 识别 AgentFamily
  2. 读取 `HIPPOCAMPUS_PRESET_SCENARIO`（默认按 agent 推导：ClaudeCode/Trae→Coding，其他→Daily）
  3. `PresetBuilder::build()` 生成 CombinedProfile（含 usage_protocol）
  4. 把 usage_protocol 注入 HippocampusMcp 实例
- [ ] `HippocampusMcp` 新增 `combined_profile` 字段 + `with_combined_profile` builder
- [ ] MCP `server_info` 的 description 字段填入 usage_protocol.instructions（兼容旧客户端）
- [ ] 测试：启动日志输出识别结果 + 应用的 preset

#### 1.5 工具描述动态覆盖（hippocampus-mcp crate）

- [ ] 11 个 tool 的 description 支持运行时按 preset 覆盖
  - `prompt` 工具：Claude Code preset 下加「【会话开始必调】」前缀
  - `archive` 工具：加「【上下文超 180K 时必调】」前缀
  - `semantic_search` 工具：加「【用户提到过去事件时调】」前缀
- [ ] rmcp 的 tool description 若不支持运行时覆盖，退化为 server_info 描述里集中说明
- [ ] 测试：不同 preset 下 tool description 不同

#### 1.6 Trae 接入 onboarding 模板

- [ ] 在 `docs/onboarding/trae.md` 写明：
  - Trae MCP 配置文件位置与格式
  - 必需环境变量（HIPPOCAMPUS_ROOT / HIPPOCAMPUS_GENERATOR_*）
  - 可选环境变量（HIPPOCAMPUS_PRESET_AGENT / HIPPOCAMPUS_PRESET_SCENARIO）
  - session_id 约定（推荐 `trae-{项目名}-{日期}`）
  - 验证步骤（启动 Trae 后 LLM 是否主动调 prompt）
- [ ] AGENTS.md 模板（项目根目录放，Trae/Cursor 自动读取）
  ```markdown
  # 本项目已接入 hippocampus 记忆库
  ## Agent 必须遵守的记忆协议
  1. 每个新会话开始 → 第一步调 mcp_hippocampus.prompt
  2. 上下文超阈值 → 调 archive 归档
  3. 用户提到过去事件 → 先调 semantic_search 再回复
  session_id 约定：trae-{项目名}-{日期}
  ```

#### 1.7 错误信息 LLM 友好化

- [ ] archive 工具的 UUID 解析错误改写：
  - 原：`UUID parsing failed: invalid character: found 't' at 0`
  - 新：`turns[].id 必须是 UUID 格式（如 7f9c1b2a-3d4e-4f5a-8a9b-0c1d2e3f4a5b），你传的 't1' 不合法`
- [ ] 所有工具错误返回结构化 JSON：`{"error": {"code": "...", "message": "...", "hint": "..."}}`
- [ ] 测试：构造非法输入验证错误消息可读

#### 1.8 SERVER_METADATA 自动填充

- [ ] MCP server 启动时把 usage_protocol.instructions 写入 SERVER_METADATA.json
  （或运行时返回给客户端，避免文件写入冲突）
- [ ] 验证：Trae 重新加载 MCP 后，LLM 能看到 hippocampus 的整体描述

**阶段 1 验收标准**：

## 关键设计决策（待确认）

### D1: 自动识别失败时如何处理？

**选项**：
- A. 降级为 Generic（默认 400K + 不输出 instructions）—— **推荐**
- B. 要求用户必须显式配置 `HIPPOCAMPUS_PRESET_AGENT`，否则启动报错

**推荐**：A，保证零配置可用，preset 是优化而非必需

### D2: scenario 默认值如何推导？

**选项**：
- A. 按 agent 推导（ClaudeCode/Trae→Coding，其他→Daily）—— **推荐**
- B. 统一默认 Coding
- C. 统一默认 Daily
- D. 必须显式配置

**推荐**：A，不同客户端有典型场景，推导降低配置成本

### D3: instructions 注入方式？

**选项**：
- A. MCP 2025-03-26 规范的 `instructions` 字段（需 rmcp 升级）
- B. 塞进 `server_info` 的 description（兼容旧客户端）—— **推荐先用 B**
- C. 两者都填（向前兼容）

**推荐**：先用 B 保证立即可用，后续 rmcp 升级后补 A

### D4: 触发规则是软约束还是硬约束？

**选项**：
- A. 软约束（usage_protocol 只是给 LLM 看的提示文本）—— **推荐**
- B. 硬约束（MCP server 主动推送 / 拦截）
- C. 混合（关键规则硬约束，其他软约束）

**推荐**：A，LLM 自主决策符合 Agent 范式，硬约束会增加 MCP 协议复杂度

### D5: Trae 多 agent 角色如何处理？

**发现**：Trae 为同一 MCP server 生成 3 个 agent 角色变体
（solo_agent_lite / search / general_purpose_task），每个都有独立的工具副本

**选项**：
- A. 忽略 agent 角色，按 Trae 客户端统一识别 —— **推荐**
- B. 按 agent 角色细分 preset（如 search 角色用 Research scenario）
- C. 让用户在环境变量中显式指定 agent 角色

**推荐**：A，先简化；如发现不同角色需要不同 preset 再考虑 B

## 风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| rmcp 不支持运行时 tool description 覆盖 | 1.5 受阻 | 退化为 server_info 集中说明 |
| rmcp 不支持 `instructions` 字段 | D3 选项 A 受阻 | 先用 server_info.description |
| Agent 客户端拉起 MCP 时的环境变量不稳定 | 1.2 识别失败 | 多信号融合 + 显式声明兜底 |
| 不同客户端 MCP 配置格式差异大 | onboarding 模板工作量大 | 按客户端逐个验证，不批量 |
| LLM 不遵守 instructions 软约束 | 工具不被主动调用 | AGENTS.md 强化提示 + 客户端 rules 文件 |
