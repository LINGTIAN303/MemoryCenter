# v2.31 路线图：Agent 上下文感知与同步归档

> **状态**：方向探讨阶段（2026-07-05）
> **目标**：让 Hippocampus 在 Agent 客户端压缩上下文时，能感知并自动归档被压缩的轮次
> **前置**：v2.30.1 已完成 archive 入参简化（[变更记录](../CHANGELOG.md)）

---

## 一、背景与问题陈述

### 1.1 用户原始设想

> 让 Agent 代理工具中的会话上下文阈值接入到 Hippocampus，
> 当 Agent 压缩上下文时，Hippocampus 自动识别并获取上下文中的轮次信息并归档。

### 1.2 核心矛盾：MCP 协议是"被动响应"模式

```
Agent 客户端（Trae/Cursor/CatPaw）     ←──主动──    Hippocampus MCP 服务端
        │                                                    │
        │ 1. 内部决定压缩上下文                                │
        │ 2. 丢弃旧轮次，保留摘要                              │
        │ 3. (Hippocampus 完全不知道发生了什么)                │
        │                                                    │
        └─→ 只有 Agent 主动调用 archive 时，Hippocampus 才能拿到数据
```

**MCP 协议的根本限制**：服务端不能主动发起请求或订阅客户端事件。Hippocampus 永远是"被调用方"，无法感知 Agent 何时压缩、压缩了什么。

### 1.3 这不是源码问题，是协议问题

即使拿到 Trae 源码（闭源，无法获取），MCP 协议也不允许服务端主动发起请求或订阅客户端事件。真正需要的是：

1. **Agent 客户端暴露压缩事件 API**（让外部能感知"即将压缩"）
2. **MCP 协议支持事件订阅**（让服务端能被动接收通知）

这两点都是**协议/平台层面的改动**，不是 Hippocampus 能单方面解决的。

## 二、三个可行方案

### 方案A：靠 AGENTS.md 规则约束（当前方案，已实施）

在 [AGENTS.md](../AGENTS.md) 里写规则：

```markdown
### 2. 上下文超阈值：调 archive 归档
当对话 token 数接近 400K（或当前 preset 的 archive_threshold）时，主动调用 archive。
```

**问题**：
- 依赖 Agent 自觉遵守规则
- Agent 不知道自己的真实 token 数（除非客户端暴露）
- Trae 自动压缩时，Agent 可能还没意识到就该压缩了
- 如果 Trae 提前压缩了，Agent 拿不到被压缩的原始内容

**适用场景**：Agent 客户端相对可控（如 Claude Code 严格遵守 AGENTS.md）

**增强方向**：
- 让 Hippocampus 在 archive 返回结果中提供估算的 token 数给 LLM
- 让 LLM 持续跟踪累计 token 数，接近阈值时主动归档
- 不依赖客户端暴露，由 Hippocampus 端承担估算责任

### 方案B：Hippocampus 提供"主动推送"接口（需 Agent 配合）

让 Hippocampus 提供一个 `pre_compress_hook` MCP 工具，Agent 客户端在压缩前调用：

```
Agent 即将压缩上下文
    ↓
调用 hippocampus.pre_compress_hook(session_id, full_context)
    ↓
Hippocampus 接收完整上下文，自动归档
    ↓
返回 OK，Agent 继续压缩
```

**前提**：Agent 客户端必须主动调用这个钩子。**Trae/Cursor 不会主动调用**（它们不知道 Hippocampus 的存在），除非：
- Agent 客户端原生集成 Hippocampus（需客户端开发者支持）
- 或通过 IDE 插件层拦截压缩事件（如 VS Code 扩展）

**适用场景**：开源 Agent 客户端（Cline/Continue/Aider 可改源码）

### 方案C：IDE 插件层拦截（最接近用户设想）

开发一个 Trae/VS Code 扩展，监听编辑器的上下文压缩事件：

```
Trae 即将压缩上下文
    ↓
VS Code 扩展拦截 onWillCompress 事件（如果 API 暴露）
    ↓
扩展调用 hippocampus MCP 的 archive 工具
    ↓
归档完成后再让 Trae 压缩
```

**前提**：Trae/VS Code 必须暴露压缩事件 API。**目前没有标准 API**。

**长期方向**：给 Trae 提 feature request，要求暴露 `onContextCompress` 事件 API。

## 三、伪钩子思路：客户端安装时主动接收订阅

### 3.1 核心思想

> 让客户端在接收/安装到 Hippocampus MCP 时，
> 在这一步上"主动"的接收 Hippocampus MCP 的"请求/订阅客户端事件"

这是一种协议层 workaround：
- 客户端安装 MCP 时，主动读取 Hippocampus 的 `usage_protocol.instructions`
- `instructions` 里包含"订阅请求"：要求客户端在压缩前调用 `pre_compress_hook`
- 客户端如果遵守，就实现了伪钩子

### 3.2 与方案A的关联

这是方案A的增强版：
- **方案A**：AGENTS.md 规则约束（依赖用户/Agent 自觉）
- **伪钩子**：MCP 安装时注入 instructions（依赖客户端解析 instructions）

MCP 协议允许服务端在 `server_info.instructions` 字段注入使用说明，Agent 客户端通常会读取并注入到 LLM 的 system prompt。Hippocampus v2.30+ 已支持自动识别 Agent 客户端并注入 `usage_protocol.instructions`。

**关键**：写好 `instructions`，让 LLM 理解"何时该调用 archive"和"如何估算 token"。

## 四、token 估算的四种途径

### 4.1 途径对比

| 途径 | 来源 | 精度 | 可用性 |
|------|------|------|--------|
| ① LLM 推算 | LLM 根据对话轮次、工具调用次数推算 | 低 | 通用 |
| ② 客户端暴露 | 客户端向 LLM 暴露用量（模糊/百分比、精准/tokens） | 中-高 | 取决于客户端 |
| ③ 用户告知 | 用户看到界面用量显示，告知 LLM | 中 | 通用 |
| ④ Hippocampus 估算 | Hippocampus 按文本长度估算，返回给 LLM | 中 | 通用 |

### 4.2 推荐路径：④ + ① 组合

**Hippocampus 端承担估算责任**：
1. 每次 archive 调用时，估算并返回累计 token 数
2. LLM 持续跟踪累计值，接近阈值时主动归档
3. 不依赖客户端暴露，通用性强

**实现要点**：
- `estimate_tokens()` 已在 v2.30.1 实现（3 char ≈ 1 token）
- 在 archive 工具的返回结果中添加 `estimated_total_tokens` 字段
- 在 `instructions` 中告知 LLM："归档后会返回累计 token 估算，超过阈值请主动归档"

### 4.3 不同 Agent 客户端的适配

| 客户端 | 上下文用量展示 | 是否暴露给 LLM | 压缩事件 API |
|--------|--------------|--------------|------------|
| Trae Solo/Work | ✅ 有进度展示 | ❓ 待确认 | ❌ 未公开 |
| Trae IDE | ❓ 未确认 | ❓ 待确认 | ❌ 未公开 |
| Trae CLI | ❓ 未确认 | ❓ 待确认 | ❌ 未公开 |
| Trae 插件 | ❌ 无 | ❌ 无 | ❌ 未公开 |
| Cursor | ✅ 有 | ❓ 待确认 | ❌ 未公开 |
| Claude Code | ✅ 有 | ✅ 部分暴露 | ❌ 未公开 |
| CatPaw | ❓ 待确认 | ❓ 待确认 | ❌ 未确认 |

#### 5.2.1 上下文压缩与 stdout/stderr（已纠正）

**实测结论（2026-07-05 纠正）**：CatPaw 的 stdout/stderr **是分离捕获**的。

Agent 实测发现：
- 写入顺序 OUT-1 → ERR-1 → OUT-2 → ERR-2
- CatPaw `run_terminal_cmd` 返回顺序：OUT-1, OUT-2, ERR-1, ERR-2
- stdout 两条全在前面，stderr 两条全在后面
- 说明 CatPaw 用了**两个独立的管道**分别捕获 stdout 和 stderr，再拼接展示（stdout 优先，无 `[stderr]` 前缀标记）

**与之前传输层污染的关系**：
- `run_terminal_cmd` 工具 vs MCP stdio 传输是两套不同的管道机制
- `run_terminal_cmd` 分离捕获，但 MCP stdio 传输下的行为未直接实测
- v2.30.1 的 `.with_ansi(false)` 仍是正确的防御性修复（即使 stderr 被错误解析，纯文本也不会被当作 JSON）
- **不再需要**向 CatPaw 提 stdout/stderr 分离的 feature request（已是分离的）

**遗留疑问**（待后续调研）：
- MCP stdio 传输下 CatPaw 是否也分离 stdout/stderr？
- 之前 `[2m2026-0...` 污染的根因是否是 tracing 初始化阶段的输出？

#### 5.2.2 Rules 系统（伪钩子方案的关键入口！）

CatPaw 支持 4 种 Rules 类型，配置在 `.catpaw/rules/{文件名}.md`：

| Rules 类型 | 触发条件 | 适用场景 | 伪钩子价值 |
|------------|---------|---------|-----------|
| **Always** | 始终生效 | 团队通用规范 | ⭐⭐⭐ 可写"主动归档规则" |
| **Model Request** | AI 模型自主判断 | 智能化处理 | ⭐⭐⭐⭐ **最佳入口**！LLM 自主判断何时归档 |
| **Auto Attached** | 文件/路径匹配 | 特定文件规则 | ⭐ 不适用 |
| **Manual** | 需 `@` 调用 | 临时规则 | ⭐ 不适用 |

**关键发现**：`Model Request` 类型的 Rule，其 `name` 和 `description` 会被发送给 AI 模型。模型根据对话内容**智能判断是否需要应用**该规则。

这意味着：
- 我们可以创建一个 `Model Request` 类型的 Rule
- Rule 的 description 告诉 LLM："当对话变长时，主动调用 hippocampus.archive 归档"
- LLM 会智能判断何时应用，**不依赖客户端暴露压缩事件**

#### 5.2.3 自定义 Agent 模式

CatPaw 支持自定义 Agent 模式，可配置：
- 自定义 System Prompt
- 启用的 Tools 工具（可勾选启用的 MCP 工具）
- 默认 Docs
- Auto-run / Auto-fix

这为 Hippocampus 提供了**第二个入口**：创建一个"记忆增强 Agent 模式"，System Prompt 中写入归档触发规则。

#### 5.2.4 兼容性

- 兼容 `CLAUDE.md` 文件（配置页开启后）
- Rules 文件可随项目代码提交到远程仓库（团队共享）

## 六、伪钩子方案：具体实施路径

### 6.1 核心思路

利用 Agent 客户端的 **Rules 系统**（CatPaw 的 `Model Request` 类型最佳），创建一个"主动归档规则"：

```
Rule 文件（.catpaw/rules/hippocampus-archive.md）
    ↓
ruleType: Model Request
    ↓
name: "Hippocampus 记忆归档"
description: "当对话变长、上下文接近阈值时，主动调用 hippocampus.archive 归档"
    ↓
LLM 收到 name + description
    ↓
LLM 智能判断："这次对话够长了，应该归档了"
    ↓
LLM 主动调用 hippocampus.archive(session_id, turns_json)
    ↓
Hippocampus 返回 estimated_total_tokens
    ↓
LLM 跟踪累计 token，下次更精准判断
```

### 6.2 双层入口设计

**第一层：客户端 Rules 文件**（用户配置一次，永久生效）
- `.catpaw/rules/hippocampus-archive.md`（CatPaw）
- `.trae/rules/hippocampus-archive.md`（Trae）
- `CLAUDE.md`（Claude Code）

**第二层：MCP server_info.instructions**（Hippocampus 自动注入）
- v2.30+ 已支持自动识别 Agent 客户端
- 在 `instructions` 字段注入使用协议
- 但**依赖客户端解析 instructions**（非所有客户端都解析）

**双层冗余**：即使客户端不解析 MCP instructions，Rules 文件也能保证规则生效。

### 6.3 token 估算的反馈循环

```
Agent 调用 archive
    ↓
Hippocampus 估算 token（estimate_tokens）
    ↓
返回结果包含：{ hook_id, archived_count, estimated_total_tokens }
    ↓
LLM 收到 estimated_total_tokens
    ↓
LLM 内部维护一个"累计 token 计数器"
    ↓
下次对话时，LLM 判断："累计已接近阈值，该归档了"
    ↓
主动调用 archive → 循环
```

**关键**：LLM 没有自我审视能力（无法感知自身 token 消耗），但 Hippocampus 的 `estimated_total_tokens` 提供了**外部感知**，让 LLM 通过外部反馈建立"token 意识"。

### 6.4 实施步骤

#### Step 1：增强 archive 返回值（v2.31 核心）

在 archive 工具的返回结果中添加 `estimated_total_tokens` 字段：
```json
{
  "hook_id": "5b30a117-...",
  "archived_count": 5,
  "estimated_total_tokens": 12500,
  "threshold": 120000,
  "threshold_ratio": 0.104,
  "suggestion": "已归档 5 轮，累计估算 12500 tokens（阈值 120K，当前 10.4%）。继续对话。"
}
```

#### Step 2：编写 Rules 模板文件

为每个客户端编写标准 Rules 模板，放在 `docs/onboarding/rules/` 目录：
- `catpaw-hippocampus-archive.md`（CatPaw Model Request 类型）
- `trae-hippocampus-archive.md`（Trae Always 类型）
- `claude-code-hippocampus-archive.md`（CLAUDE.md 片段）

#### Step 3：强化 AGENTS.md / instructions

在 AGENTS.md 中明确：
- 何时该归档（基于 estimated_total_tokens 反馈）
- 如何跟踪累计 token（LLM 内部维护计数器）
- 兜底策略（每 N 轮主动归档）

## 七、关于 LLM 查询自身 token 用量

### 7.1 用户期望

> 你可以试试看能不能看到当前上下文用了多少 token

### 7.2 现实情况

**LLM 无法直接感知自身 token 消耗**。这是 LLM 的固有限制：
- 我没有"自我审视"能力
- 无法访问自身的 token 计数器
- 无法感知上下文窗口的占用情况

### 7.3 解决方案：外部反馈循环

虽然 LLM 无法自检，但可以通过**外部工具反馈**建立 token 意识：

1. **Hippocampus archive 返回 estimated_total_tokens**（已规划 v2.31）
2. **LLM 在每次 archive 后记录累计值**
3. **下次对话时 LLM 引用上次归档返回的 token 数**

这是一个"伪 token 感知"机制——LLM 通过外部反馈建立内部计数器，而非直接感知。

### 7.4 4 种 token 估算途径的再评估

| 途径 | 可行性 | 精度 | 实施难度 |
|------|--------|------|---------|
| ① LLM 推算（轮次/工具调用次数） | ✅ 可行 | 低 | 低 |
| ② 客户端暴露给 LLM | ❌ Trae/CatPaw 不暴露 | - | - |
| ③ 用户告知 | ✅ 可行 | 中 | 依赖用户 |
| ④ Hippocampus 估算返回 | ✅ 可行（推荐） | 中 | 中 |

**推荐**：④ + ① 组合
- ④ 提供客观基准（Hippocampus 估算）
- ① 提供主观判断（LLM 根据对话复杂度调整）
- 两者结合，让 LLM 在没有客户端暴露的情况下也能近似感知 token 用量

## 八.5、压缩后接续机制的实锤发现（2026-07-06）

### 8.5.1 触发场景

用户在 v2.31 讨论中提出的关键问题：

> "Trae 自动压缩被触发后，客户端是怎么向你发起请求来接续上下文被压缩后的？"

通过 LLM 第一视角实证（本会话内已发生至少 1 次自动压缩），确认了 Trae 压缩后的接续机制。

### 8.5.2 LLM 实际接收的 3 层信息

**不是原始对话轮次**，而是 Trae 重组的 3 层结构：

#### 第 1 层：标记消息（告知"这是延续"）

```
This session continues a previous conversation that lost its context.
A summary has been provided. Please proceed with the task from this point,
using the summary as your guide to avoid repeating any completed steps.
Here is the user most recent query:
[用户最新消息原文]
The conversation is summarized below:
```

**关键点**：Trae 直接告知"上下文丢了，给你 summary 凑合用"。但**不告知**压缩发生时间、原始轮次数、丢弃了哪些内容。

#### 第 2 层：结构化 Summary（11 个章节，LLM 生成）

```
1. Primary Request and Intent          ← 用户请求与意图演变
2. Key Technical Concepts              ← 技术概念关键词
3. Files and Code Sections             ← 相关文件路径 + 代码片段（带行号）
4. Errors and fixes                    ← 已解决的错误
5. Problem Solving                     ← 问题状态
6. All user messages                   ← 用户消息简短摘要
7. Pending Tasks                       ← 待办任务
8. Current Work                        ← 当前工作状态
9. Working on Plan/Spec                ← 正在做的计划
10. Optional Next Step                 ← 建议下一步
11. Conversation Language              ← 对话语言
```

#### 第 3 层：附加 Recent files + Pending todos

```
Recent important skills and files from this conversation are included below.
- Recent file: handlers.rs [带行号的完整代码]
- Recent file: lib.rs [部分代码]
- Recent file: main.rs [部分代码]
- Recent file: Cargo.toml [完整内容]
- Recent file: AGENTS.md [完整内容]

Pending todos:
- [继承的待办列表]
```

### 8.5.3 关键发现

1. **LLM 接收的不是原始轮次**：看不到原始问题文本、用户回答措辞、完整工具调用记录
2. **Trae 压缩是有损压缩 + 结构化重组**：用 LLM 重新生成 11 章节结构化 summary
3. **LLM 不知"被压缩了"**：标记消息只说"continues a previous conversation"，无法区分"刚压缩"vs"几小时前压缩"
4. **Recent files 机制可利用**：Trae 自动保留"工作集"文件，Hippocampus 归档时可参考类似机制
5. **Summary 结构稳定可预测**：11 章节格式可被 Rules 识别

### 8.5.4 实锤的"无法解决缺口"

之前分析的两个缺口，本次实证确认：

- **缺口 1（LLM 不知被压缩）**：✅ 实锤。LLM 看到的就是"summary + 新消息"，与正常连续对话无本质区别
- **缺口 2（未归档内容永久丢失）**：✅ 实锤。如果 LLM 压缩前没主动 archive，原始轮次被 Trae 丢弃后永远拿不回

### 8.5.5 四个推进方向（待 v2.31 BUG 修复后深入）

#### 方向 1：利用标记消息触发 prompt（✅ 已实施，2026-07-06）

**实施内容**：在 `.trae/rules/hippocampus-archive.md` 中新增「压缩后行为协议」章节，AGENTS.md 同步更新。

**已实现能力**：
1. Marker Message 识别：固定文本 `This session continues a previous conversation that lost its context.` 触发协议
2. 强制流程（4 步）：
   - 步骤1：归档压缩前未持久化的轮次
   - 步骤2：调用 `prompt` 拉取 hippocampus 一手记忆
   - 步骤3：交叉校准 Summary 第8章节"Current Work"与 hippocampus 记忆
   - 步骤4：执行「Next Step 决策协议」
3. Next Step 决策协议（3 种情况）：
   - Pending todos 有 in_progress → 优先继续该任务，忽略 Next Step 建议
   - Pending todos 有 pending → 按优先级执行，Next Step 仅作参考
   - Pending todos 为空 → 执行 Next Step 但需先向用户确认

**优先级原则**：
- hippocampus 记忆 > Trae Summary（一手源数据 > 二手信息）
- Pending todos 状态 > Summary 第10章节建议（实际状态 > 推荐建议）
- in_progress 任务必须从断点继续，禁止重复提问已完成决策

**验证方式**：下次 Trae 自动压缩时，LLM 应自动执行上述 4 步流程，不再"再次提问已完成决策"。

**风险遗留**：
- Marker Message 格式若随 Trae 版本变化，需同步更新 Rules
- 严重不一致场景仍需向用户确认（无法完全自动化）

#### 方向 2：深入分析 Trae 压缩机制（实证研究）

让 LLM 多触发几次压缩，观察：
- summary 的 11 章节结构是否稳定
- Recent files 的选择规则是什么（最近修改？最长？被引用最多？）
- 不同场景下保留内容是否有差异（编码 vs 写作 vs 讨论）
- 压缩前后 LLM 行为差异

用实证数据指导方案设计。
